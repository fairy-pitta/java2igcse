# Java to IGCSE Pseudocode Converter - プロジェクトルール

## 🎯 プロジェクトの目的

JavaのソースコードをIGCSE Computer Science仕様のpseudocodeに変換するライブラリを開発する。
変換時の一貫性と正確性を確保するため、以下のルールを厳守する。

## 📝 コード変換の基本ルール

### インデント処理

1. **基本インデント**
   - 3スペースを基本単位とする（IGCSE推奨）
   - インデントレベルは最大10階層まで
   - 空行のインデントは保持しない

2. **インデント対象**
   - 制御構造（IF, WHILE, FOR等）のブロック内
   - PROCEDURE/FUNCTIONの本体
   - CLASSの本体
   - CASE文の各ケース

3. **インデント禁止**
   - ファイル先頭のDECLARE文
   - トップレベルの型定義
   - エラーメッセージ

### ネスト構造の処理

1. **再帰防止**
   - 最大ネスト深度: 10レベル
   - 循環参照検出機能の実装
   - 深度超過時は適切なエラーメッセージ

2. **親子関係の追跡**
   - 各ノードに一意のID付与
   - 親ノードの参照を保持
   - 変換履歴の記録

### IF-ELIF-ELSE構造

1. **基本構造**
```pseudocode
IF condition1 THEN
   statement1
ELSE
   IF condition2 THEN
      statement2
   ELSE
      statement3
   ENDIF
ENDIF
```

2. **ネストルール**
   - ELSE IF は必ずネストしたIF-THEN-ELSE-ENDIFとして展開
   - 各条件ブロックは独立したスコープとして処理
   - ENDIFの数は正確に制御

## 🧪 テスト規約

1. **テスト実行**
   - ウォッチモードは使用禁止
   - `npm test`ではなく個別のテストファイルを実行

2. **テストケース設計**
   - 各変換ルールに対する単体テスト
   - エッジケースの網羅
   - 複雑なネスト構造のテスト
   - インデントの正確性検証

3. **テスト環境**
   - Node.js 18以上
   - TypeScript 5.0以上
   - Vitest

## 🛠️ 開発規約

1. **コードスタイル**
   - ESLint + Prettierの設定に従う
   - TypeScriptの厳格モード使用
   - 関数の型定義必須

2. **エラー処理**
   - カスタムエラークラスの使用
   - エラーメッセージの日本語対応
   - スタックトレースの保持

3. **ドキュメント**
   - JSDoc形式のコメント
   - README.mdの定期更新
   - 変更履歴の記録

## 🔍 コードレビュー基準

1. **必須確認項目**
   - インデント処理の正確性
   - ネスト構造の適切な処理
   - IF-ELIF-ELSE構造の正確な変換
   - テストケースの十分な網羅

2. **品質基準**
   - テストカバレッジ90%以上
   - 循環的複雑度10未満
   - コードの重複率5%未満

## ⚠️ 制限事項

1. **非対応機能**
   - ジェネリクス
   - ラムダ式
   - アノテーション
   - マルチスレッド

2. **変換制限**
   - 最大ファイルサイズ: 1MB
   - 最大行数: 1000行
   - 最大クラス数: ファイルあたり1つ

## 📈 パフォーマンス要件

1. **実行速度**
   - 1000行未満のファイル: 1秒以内
   - 変換処理のタイムアウト: 5秒

2. **メモリ使用**
   - 最大ヒープサイズ: 512MB
   - 一時オブジェクト最小化

## 🔄 バージョン管理

1. **ブランチ戦略**
   - main: 安定版
   - develop: 開発版
   - feature/*: 機能追加
   - fix/*: バグ修正

2. **コミットルール**
   - プレフィックス必須（feat:, fix:, etc）
   - 日本語コミットメッセージ
   - 関連イシュー番号の記載
